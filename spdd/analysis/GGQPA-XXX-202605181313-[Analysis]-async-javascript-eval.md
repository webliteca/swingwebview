# SPDD Analysis: Future-Returning JavaScript Eval API

## Original Business Requirement

The full story `[User-story-3]async-javascript-eval.md` is preserved verbatim in `requirements/[User-story-3]async-javascript-eval.md` and is incorporated by reference. The headline capability:

> Add `CompletableFuture<String> evalAsync(String js)` to the standalone `WebView` (`ca.weblite.webview.WebView`, canvas-2) and the embedded `WebViewComponent` (`ca.weblite.webview.swing.WebViewComponent`, canvas-5) with concrete implementations in `WebViewHeavyweightComponent` (canvas-6) and `WebViewLightweightComponent` (canvas-7). The future completes with the JSON-serialized result of evaluating `js` (with `undefined` → `null`), awaits returned Promises, completes exceptionally on synchronous JS throws / Promise rejections / lifecycle violations, and (for the embedded surfaces) fires its continuations on the Swing EDT. Implementation is Java + an injected JS shim — no JNI additions — because native `webview_eval` / `webview_embed_eval` / `webview_offscreen_eval` are all `void`-returning.

All 13 Acceptance Criteria and the four Non-Functional Expectations from the story are in scope of this analysis.

## Domain Concept Identification

### Existing Concepts (from codebase)

- **`WebView`** (`src/ca/weblite/webview/WebView.java`, canvas-2): standalone in-process wrapper that owns its native peer and its own event loop. Holds `bindings: Map<String, JavascriptCallback>` and `heap: ArrayList` (anti-GC anchor for live callbacks). Today exposes `eval(String js)` (fire-and-forget) and `addJavascriptCallback(String, JavascriptCallback)`. The new `evalAsync` lives alongside `eval`.
- **`WebViewComponent`** (`src/ca/weblite/webview/swing/WebViewComponent.java`, canvas-5): abstract Swing `JComponent` that declares the embedding API surface. Already declares abstract `eval(String)` and `addJavascriptCallback(String, JavascriptCallback)`. Already publishes the constant `RESERVED_BINDING_PREFIX = "__webview_"` (line 53) and runtime-enforces it in both subclasses' `addJavascriptCallback` overrides (rejects with `IllegalArgumentException`).
- **`WebViewHeavyweightComponent`** (canvas-6): concrete subclass backed by `EmbeddedWebView`. Delegates `eval` / `addJavascriptCallback` / `dispatch` to its `embedded` field; the `embedded` field is created lazily in `createPeer()` when the underlying `Canvas` becomes displayable.
- **`WebViewLightweightComponent`** (canvas-7): concrete subclass backed by `OffscreenWebView`. Delegates similarly; the `engine` field is created lazily in `addNotify()`.
- **`EmbeddedWebView`** (`src/ca/weblite/webview/EmbeddedWebView.java`): low-level wrapper around the heavyweight native peer. Exposes `eval(String)`, `addJavascriptCallback(String, WebView.JavascriptCallback)`, `dispatch(Runnable)`, `dispose()`. Holds its own `heap` and `bindings` map plus a `peer` long. `checkAlive()` throws `IllegalStateException` when `peer == 0`.
- **`OffscreenWebView`** (`src/ca/weblite/webview/OffscreenWebView.java`): low-level wrapper around the lightweight native peer. Same shape as `EmbeddedWebView`.
- **`WebViewNative`** (`src/ca/weblite/webview/WebViewNative.java`): JNI surface. Three eval entry points (`webview_eval`, `webview_embed_eval`, `webview_offscreen_eval`) all return `void`. Three bind entry points (`webview_bind`, `webview_embed_bind`, `webview_offscreen_bind`) register a `WebViewNativeCallback` against a JS function name and pass it the JSON-encoded argument tuple from the JS side. Three dispatch entry points (`webview_dispatch`, `webview_embed_dispatch`, `webview_offscreen_dispatch`) marshal a `Runnable` onto the native UI thread.
- **`WebViewNativeCallback`**: functional interface with `void invoke(String arg, long wv)` invoked by the native bind machinery.
- **`ConsoleDispatcher`** (`src/ca/weblite/webview/ConsoleDispatcher.java`): per-component fan-out hub for the `__webview_console__` reserved binding. Its `SHIM_JS` constant base64-encodes the payload on the JS side; its `dispatch(String rawJson)` method extracts the first arg via `extractFirstArg` (an `indexOf`-only parser tuned to the `{"name":..., "seq":..., "args":[...]}` envelope produced by the embed bind machinery) and then base64-decodes it. Fans out to listeners on the Swing EDT via `SwingUtilities.invokeLater`. This is the closest existing analogue to what the new resolver needs.
- **`WebViewMouseDispatcher`** (`src/ca/weblite/webview/WebViewMouseDispatcher.java`): same pattern, different reserved channel (`__webview_dom_event`). Confirms that "one dispatcher per internal channel, registered eagerly at peer-create time" is the established pattern.
- **`RESERVED_BINDING_PREFIX`**: published public constant `"__webview_"`. Both `WebViewHeavyweightComponent.addJavascriptCallback` (line 153) and `WebViewLightweightComponent.addJavascriptCallback` (line 416) throw `IllegalArgumentException` when a user passes a name starting with this prefix. Internal channels register their reserved bindings by going *directly through the engine* (`embedded.addJavascriptCallback(...)`, `engine.addJavascriptCallback(...)`) rather than through the component's overridden method — see `WebViewHeavyweightComponent.createPeer` (line 412) and `WebViewLightweightComponent.addNotify` (line 211).

### New Concepts Required

- **`EvalDispatcher`** (new class, parallel to `ConsoleDispatcher` and `WebViewMouseDispatcher`): a per-engine helper that owns the in-flight `id → CompletableFuture<String>` map, the JS wrapper template, the resolver-callback parsing, the request-id sequencer (`AtomicLong`), and the dispose path that completes all pending futures exceptionally. Same lifecycle semantics as the existing dispatchers (per-instance, registered eagerly at peer creation, survives the document's navigation cycle within one peer lifetime). The dispatcher's public API is roughly `CompletableFuture<String> evalAsync(String js, EvalSink sink)` (where the `EvalSink` is the engine-specific wrapper around `webview_eval` / `webview_embed_eval` / `webview_offscreen_eval`) plus `void disposeAllPending()`.
- **The reserved resolver binding** (name TBD — see Risk R1): the JS-side global function that `EvalDispatcher.SHIM_JS` posts envelope `{id, ok, value/error}` to. Registered once per peer lifetime, alongside the existing `__webview_console__` and `__webview_dom_event` bindings.
- **`evalAsync(String)` public method**: lives on `WebView`, `WebViewComponent` (abstract), `WebViewHeavyweightComponent`, `WebViewLightweightComponent`. The two subclass implementations delegate into the dispatcher held by their backing engine wrapper (`EmbeddedWebView` / `OffscreenWebView`); the standalone `WebView` holds its dispatcher directly.

### Conceptual Relationships

- One `EvalDispatcher` instance per engine wrapper instance (`WebView` itself, each `EmbeddedWebView`, each `OffscreenWebView`). Lifecycle is tied to the engine's `peer` field — the dispatcher is constructed when its owning wrapper is constructed and torn down (pending futures completed exceptionally) when the wrapper is disposed.
- The dispatcher is the *holder* of in-flight state; the engine wrapper is the *delivery channel* (it owns the native eval call). The dispatcher delegates the actual eval call back to its owner via an `EvalSink` interface, matching the `FlagSink` pattern that `WebViewMouseDispatcher` already uses to decouple itself from the engine.
- The resolver binding is owned by the dispatcher (the dispatcher provides the `WebView.JavascriptCallback` lambda; the engine wrapper registers it with the native peer). No new JNI is involved.
- The `WebViewComponent` subclasses route their `evalAsync(String js)` calls to the engine wrapper's `evalAsync(String js)`, which routes to its dispatcher. When the future eventually completes inside the dispatcher, embedded-side completions are marshalled onto the EDT before invoking continuations; standalone `WebView` completions are not.

### Key Business Rules

- **Lifecycle invariant**: `evalAsync` on a dead/un-shown peer must return an already-failed future with `IllegalStateException`. No native call may be issued in that case. Governs `WebView` (when `peer == 0`), `EmbeddedWebView` (when disposed), `OffscreenWebView` (when disposed), and both Swing components (when not yet attached / already detached).
- **Per-request isolation**: every `evalAsync` call gets its own monotonically increasing id; concurrent calls do not interfere; results are routed strictly by id. Governs the dispatcher's in-flight map.
- **Promise awaiting**: any return value from the user's JS is wrapped in `Promise.resolve(...)` before being JSON.stringify'd, so a returned Promise's settled value (not the Promise itself) is what crosses the binding. Governs the JS shim.
- **Error symmetry**: synchronous JS throws, Promise rejections, and `JSON.stringify` failures (DOM nodes, circular refs) are all caught on the JS side and surfaced as exceptional Java futures. The Java side does not distinguish between them — the cause is always a single exception type carrying the JS-side message. Governs the dispatcher's result-routing path.
- **Threading asymmetry**: embedded surfaces complete on the Swing EDT (matching `ConsoleListener` and `WebViewMouseListener` conventions); the standalone `WebView` completes on whatever native UI thread fired the binding callback (no Swing involved, no other thread to marshal to). Governs which dispatcher is used and whether it wraps its `complete` in a `SwingUtilities.invokeLater`.
- **Dispose drains pending futures**: every still-pending future at dispose time completes exceptionally with `IllegalStateException(<wrapper-type> + " disposed")` so `.get()`-waiting callers wake up. Governs `dispose()` on `EmbeddedWebView` / `OffscreenWebView` and the equivalent teardown path on `WebView`.
- **Reserved-binding contract**: the resolver binding is owned by this story and registered through the engine wrapper's direct bind (bypassing the component-level reserved-prefix guard), just like the existing console / mouse channels. Callers cannot register a binding of the same name through `addJavascriptCallback`.

## Strategic Approach

### Solution Direction

Mirror the existing two-internal-channel pattern (`ConsoleDispatcher` + `WebViewMouseDispatcher`) for a third channel — eval result routing.

For each engine wrapper (`WebView`, `EmbeddedWebView`, `OffscreenWebView`):

1. **Construction**: instantiate a per-wrapper `EvalDispatcher` whose state is an `AtomicLong nextId`, a `ConcurrentHashMap<Long, CompletableFuture<String>> pending`, and (for the embedded variants) a `boolean marshalToEdt = true` flag. The `WebView` variant sets `marshalToEdt = false`.
2. **Peer creation** (after the native peer is alive, alongside the existing internal-channel registrations):
   - Install the dispatcher's `SHIM_JS` via `addOnBeforeLoad` (it base64-encodes the payload before posting). For the standalone `WebView`, this means appending the shim to `onBeforeLoad` and replaying it as part of `show()`, exactly like the user's `addOnBeforeLoad` snippets are replayed.
   - Register the resolver binding by calling the engine wrapper's *direct* `addJavascriptCallback` (the embedded subclasses' `addJavascriptCallback` overrides reject `__webview_*` names; the engine wrappers themselves do not). For the standalone `WebView`, the existing `addJavascriptCallback(String, JavascriptCallback)` happens to have no reserved-prefix check (canvas-2 does not own one), so a direct call is also fine there.
   - The resolver callback parses the bind envelope, extracts the base64 payload, decodes it, parses the `{id, ok, value}` / `{id, ok:false, error}` envelope (again by `indexOf` — no full JSON parsing needed because the JS wrapper guarantees the shape), looks up the future, removes it from `pending`, and completes it.
3. **`evalAsync(String js)` call path** (callable from any Java thread):
   - If the engine is not alive, return `CompletableFuture.failedFuture(new IllegalStateException(...))` (Java 8 spelling: `CompletableFuture<String> f = new CompletableFuture<>(); f.completeExceptionally(...); return f;`).
   - Assign `id = nextId.incrementAndGet()`, allocate `f = new CompletableFuture<>()`, insert `pending.put(id, f)` (BEFORE issuing the eval call so the binding callback can never lose the race — see Risk R4).
   - Build the wrapped JS snippet by inlining the id literal into the wrapper template. Call the engine's native eval with the wrapped snippet.
   - Return `f`.
4. **Dispose path**: walk `pending`, complete every entry exceptionally with `IllegalStateException(<wrapper-type> + " disposed")`, then clear the map. Done before the heap clears so the dispatcher's resolver callback (anchored in the heap) can't be GC'd mid-completion.

Public API delegation:

- `WebView.evalAsync(String)` → direct call to its `EvalDispatcher`.
- `WebViewComponent.evalAsync(String)` is abstract.
- `WebViewHeavyweightComponent.evalAsync(String)` delegates to `embedded.evalAsync(String)` (new public method on `EmbeddedWebView`).
- `WebViewLightweightComponent.evalAsync(String)` delegates to `engine.evalAsync(String)` (new public method on `OffscreenWebView`).

The story's "Scope Out: adding `evalAsync` to `EmbeddedWebView` / `OffscreenWebView` as a public surface" needs a clarification: the *story-visible* public surface (the one `WebView` and `WebViewComponent` callers see) lives on the high-level classes. But the engines do need a method to delegate to (whether named `evalAsync` and made public, or named something else and made package-private). Recommendation: add `evalAsync(String): CompletableFuture<String>` as public on both engines, mirroring how `eval(String)` is already public. The "public surface" the story is excluding is the *documentation surface* — callers should use `WebViewComponent`, not `EmbeddedWebView` / `OffscreenWebView` directly, which is the same convention as for `eval`.

### Key Design Decisions

- **Reserved binding name**: `__webview_eval_result__` (NOT `__swv_evalResult__` as proposed in the story).
  - Trade-offs: changing the name doesn't reduce risk — it just aligns with the existing codebase convention. Using a `__swv_*` name would leave a hole in the runtime-enforcement check (which only rejects `__webview_*` prefixes), meaning callers could accidentally register a binding of the same name through `addJavascriptCallback` and break `evalAsync`.
  - Recommendation: use `__webview_eval_result__`. Rationale: matches the existing `RESERVED_BINDING_PREFIX = "__webview_"` enforcement, matches the two existing internal channels (`__webview_console__`, `__webview_dom_event`), and gets collision protection on the embedded subclasses for free.
- **Binding-register timing**: eager at peer creation, not lazy on first `evalAsync` call.
  - Trade-offs: lazy is one fewer always-on JS shim per peer; eager removes a thread-safety race where two concurrent `evalAsync` calls would otherwise race to register the binding. The shim is tiny (a few lines) and the existing console + mouse channels already register eagerly, so eager has zero net cost.
  - Recommendation: eager, alongside the existing `ConsoleDispatcher.SHIM_JS` and `WebViewMouseDispatcher.SHIM_JS` registrations. This is a slight refinement to the story's "lazily registers... on first use" wording — the Canvas will document it.
- **JS-side payload encoding**: base64-encode the JSON envelope before posting through the binding.
  - Trade-offs: alternative is to JSON-escape-parse the envelope on the Java side, which means writing (or borrowing) a real JSON parser. The codebase has no external JSON dependency (only JUnit 4.12 is in `pom.xml`), and both existing internal dispatchers solve this exact problem by base64-encoding.
  - Recommendation: base64-encode. The dispatcher's `dispatch(rawJson)` method then mirrors `ConsoleDispatcher.dispatch` almost line-for-line.
- **Result format**: the Java future yields the JSON.stringify'd value as a `String`.
  - Trade-offs: callers have to parse the JSON to get a typed value. Alternative is to return `Object` (parsed) or a typed wrapper, but both require a JSON parser on the Java side which the codebase doesn't have.
  - Recommendation: stay with the story's `CompletableFuture<String>`. Callers using Jackson / Gson / etc. parse it themselves. A future story can add typed convenience wrappers if there's demand.
- **Threading on completion** (the asymmetry between AC10 and AC11):
  - Embedded surfaces: complete the future inside a `SwingUtilities.invokeLater` so any registered `.thenAccept` continuation runs on the EDT, matching the existing `ConsoleListener` and `WebViewMouseListener` contracts.
  - Standalone `WebView`: complete the future on whatever native thread the binding callback fires on. No Swing is in the picture; the standalone `WebView` owns the main thread for its event loop, and dispatching to a different thread would introduce a hop that has no obvious destination. Document this in canvas-2.
  - The two dispatchers can share a single class with a `boolean marshalToEdt` flag set at construction time. The standalone wrapper passes `false`; both embedded wrappers pass `true`.
- **Future on dispose**: complete pending futures exceptionally BEFORE the native heap clears.
  - Trade-offs: alternative is to leave them dangling, but that violates AC NFR #4 and leads to host apps hanging on `.get()`.
  - Recommendation: in `EmbeddedWebView.dispose` / `OffscreenWebView.dispose` (and the equivalent place for `WebView` — there is no current dispose path; calling `evalAsync` after the run loop returns will need to be considered, see Risk R8), call `evalDispatcher.disposeAllPending()` first, then proceed with the existing teardown.
- **Cancellation**: `CompletableFuture.cancel(true)` marks the future cancelled but does not abort the in-page JS.
  - Trade-offs: aborting the in-page JS would require a JS-side cancellation token plumbed through every wrapped snippet, which is out of scope.
  - Recommendation: leave the entry in `pending`; when the binding callback eventually fires, it finds a cancelled future and silently drops the result. Document in the Canvas.

### Alternatives Considered

- **Add a JNI `webview_eval_with_callback` that takes a Java callback and rolls the round-trip into native code**: rejected because it requires touching three native backends (`src_c/webview.c` for the zserge engine, plus the two custom embed/offscreen layers), and the upstream zserge engine is shared with the broader webview ecosystem — changing it complicates upgrades. The Java + injected-JS approach is the same approach the existing console capture and mouse-event channels take.
- **Place the in-flight map on `WebViewComponent` itself, shared across both subclasses**: rejected because the standalone `WebView` has no `WebViewComponent` and would need its own copy anyway. Putting one dispatcher per engine wrapper (three places total) is symmetric and matches the established pattern.
- **Reuse the public `addJavascriptCallback` surface to register the resolver binding from `evalAsync`**: rejected because `WebViewHeavyweightComponent` and `WebViewLightweightComponent` reject `__webview_*` names. The existing dispatchers bypass the guard by going directly through the engine wrapper's `addJavascriptCallback`; the resolver follows that same path.
- **Lazy binding registration on first `evalAsync` call**: rejected in favor of eager registration at peer creation (see Key Design Decisions). The story's "lazily registers... on first use" wording is a small refinement opportunity, not a hard constraint.

## Risk & Gap Analysis

### Requirement Ambiguities

- **R1 — Reserved binding name does not match codebase convention**: The story proposes `__swv_evalResult__`. The codebase has already established `__webview_` as THE reserved prefix (`WebViewComponent.RESERVED_BINDING_PREFIX`), with runtime enforcement that throws `IllegalArgumentException` when user code attempts to register a `__webview_*` name. A `__swv_*` name would NOT be caught by that guard, meaning a user who innocently calls `addJavascriptCallback("__swv_evalResult__", ...)` would silently break `evalAsync`. **Recommendation**: rename the resolver binding to `__webview_eval_result__` in the Canvas. The Canvas will adopt this rename automatically; the story's AC9 ("Reserved resolver binding does not collide with user bindings") still passes — in fact, it passes *more strongly* because the runtime rejection kicks in.
- **R2 — Engine `evalAsync` visibility**: The story's "Scope Out" excludes `evalAsync` from the public surface of `EmbeddedWebView` / `OffscreenWebView`, but the Swing component implementations need a method to delegate to. Two readings: (a) the engine method is package-private and not part of the documented surface; (b) the engine method is public and just happens not to be the "recommended" surface, parallel to how `eval` is public on the engines but the docs steer callers to the component. **Recommendation**: reading (b) — public on the engines, mirroring `eval`. The Canvas will document this.
- **R3 — Resolver registration timing**: Story says "lazily registers... on first use". Eager registration at peer creation removes a thread-safety race and matches the two existing internal channels. **Recommendation**: eager (see Key Design Decisions). The Canvas will document the eager pattern; this is a strengthening of the story, not a contradiction.

### Edge Cases

- **E1 — `evalAsync` called between `WebView.show()` returning and the run loop running**: The standalone `WebView.show()` assigns `peer = WebViewNative.webview_create(...)` and then immediately runs the event loop, which blocks until the window closes. `show()` is one-shot. Callers who invoke `evalAsync` from a different thread between peer creation and the run loop starting are fine (the binding is already registered by then, since the resolver is part of the `onBeforeLoad` replay in `show()`). Callers who invoke it AFTER `show()` returns (i.e., after the window has been closed) hit a dead peer; current `WebView.java` does not zero out `peer` after the run loop returns (it just stops being valid). **Recommendation**: zero out `peer` after `webview_run` returns AND complete all pending futures exceptionally at that point, so `evalAsync("...")` after window-close returns a failed future instead of crashing the JVM with a SIGSEGV on a freed native handle. This is also a small fix to canvas-2's known limitation about post-`show()` state.
- **E2 — Page navigation while futures are in-flight**: When the page navigates, any in-flight JS evaluating in the old document never resolves (the document is gone). Today's `eval` has the same problem silently (it's fire-and-forget so there is no future to leak). For `evalAsync`, the future will stay pending forever unless something cleans it up. **Recommendation**: document as a known limitation in the Canvas. Callers who need a deadline can `.orTimeout(...)` the returned future themselves (Java 9+) or wrap with `.get(timeout, unit)`. A future story could add an internal navigation-listener that drains pending futures on every navigation, but that requires per-engine plumbing not currently in place.
- **E3 — Multiple `evalAsync` calls in rapid succession**: The dispatcher's `AtomicLong` sequencer guarantees unique ids; concurrent calls do not interfere. Covered by AC8.
- **E4 — `evalAsync("")` (empty snippet)**: The IIFE form `(function(){ })()` returns `undefined`, the inner `Promise.resolve(undefined)` resolves immediately, the future completes with `"null"`. Same path as AC3. No special handling needed.
- **E5 — `evalAsync(null)`**: Calling the native eval with a null `String` arg is a JNI no-no on at least some engines. **Recommendation**: `Objects.requireNonNull(js, "js")` at the top of `evalAsync`, throw `NullPointerException` synchronously (NOT a failed future — null `js` is a programming error, not a runtime condition).
- **E6 — Snippet that itself calls the resolver binding**: e.g. `evalAsync("window.__webview_eval_result__('garbage'); return 1;")`. The resolver callback will be invoked with a payload that doesn't parse as `{id, ok, value}`. **Recommendation**: the dispatcher's parse path silently drops malformed payloads (mirroring how `ConsoleDispatcher.dispatch` silently drops malformed wire formats). The legitimate result for `id=N` still arrives via the wrapper's normal path. Document as part of the resolver-binding contract.
- **E7 — Snippet whose Promise never settles**: pending forever, same as E2. Same recommendation.
- **E8 — Snippet that mutates `window.__webview_eval_result__`**: e.g. `evalAsync("window.__webview_eval_result__ = function(){}; return 1;")`. The wrapper's `then` calls `window.__webview_eval_result__(...)` and routes to user-controlled JS instead of the native binding. **Recommendation**: the wrapper template should cache a reference to the binding before invoking the user snippet, e.g. `var __sink = window.__webview_eval_result__; ... __sink(...);`. This makes the binding reference robust against user-side mutation of `window`. The Canvas will spec the wrapper template this way.

### Technical Risks

- **R4 — Race between binding callback firing and `pending.put`**: If the wrapper template is evaluated synchronously by the engine before `pending.put(id, f)` completes, the binding callback could find no entry and silently drop a legitimate result. **Mitigation**: insert into `pending` BEFORE issuing the `webview_eval` call. The engine's eval is asynchronous (documented in `WebViewNative.webview_eval` Javadoc: "Evaluation happens asynchronously"), but defending against a synchronous-eval engine costs nothing.
- **R5 — Heap retention vs. dispose ordering**: The resolver `WebViewNativeCallback` lambda is anchored in the engine wrapper's `heap` list to prevent GC. On dispose, the existing `EmbeddedWebView.dispose` clears `heap` AFTER calling `webview_embed_destroy`. If a pending JS callback fires during the destroy call (race), it would find the resolver still alive (heap not cleared) but the dispatcher's `pending` map already drained (because step 3 of dispose happens first). The callback would silently drop, which is the desired behavior. **Mitigation**: ensure `disposeAllPending` runs first; the heap clear remains where it is.
- **R6 — Java 8 + no JSON dependency**: The codebase has only JUnit 4.12. `CompletableFuture` is Java 8 (available). `CompletableFuture.failedFuture(...)` is Java 9 (NOT available); the analysis uses the Java 8 idiom (`new CompletableFuture<>()` + `completeExceptionally`). No JSON library; the dispatcher uses `indexOf`-based slicing on a known-shape payload, base64-decoded from the JS side. Same pattern as `ConsoleDispatcher`.
- **R7 — Listener exception in continuation does not break the dispatcher**: a user's `.thenAccept(s -> { throw new RuntimeException(); })` does NOT affect `pending` or other futures — `CompletableFuture` semantics isolate continuation exceptions to the dependent stage. No additional defensive code needed in the dispatcher.
- **R8 — Standalone `WebView` post-`show()` state**: `show()` is one-shot; after the run loop returns the peer is invalid but `peer` field is non-zero (current behavior, per canvas-2 known limitations). Without intervention, post-close `evalAsync` calls would issue a native call on a freed handle. **Mitigation**: in `WebView.show()`, after `webview_run` returns, set `peer = 0` and call `evalDispatcher.disposeAllPending()`. This is a small extension to the existing show-method contract and is congruent with the story's NFR #4 for the embedded case.
- **R9 — Marshaling continuations to the EDT when no Swing is loaded**: the standalone `WebView` does NOT have Swing on the classpath as a hard requirement. The dispatcher must NOT touch `SwingUtilities` for the standalone case. **Mitigation**: the standalone wrapper instantiates its dispatcher with `marshalToEdt = false`; the dispatcher branches on the flag.
- **R10 — Lightweight component's snapshot timer thread vs. binding-callback thread**: the lightweight component runs a Swing `Timer` to repaint the offscreen pixels. The binding callback fires on the GTK pump thread. The dispatcher's `SwingUtilities.invokeLater` correctly hops to the EDT before completing the future, so the painting timer and the future-continuation never race on Swing state. No additional mitigation needed.
- **R11 — Init-script ordering for the standalone `WebView`**: the existing `WebView.show()` replays `onBeforeLoad` scripts AFTER setting bounds and BEFORE bindings, then navigates. The eval-shim needs to land in `onBeforeLoad` so it runs on every navigation; the resolver binding needs to be registered as part of the bindings replay. **Mitigation**: the standalone `WebView` will need a small extension to its `show()` to install the eval-shim and the resolver binding before the `webview_run` call, parallel to how the heavyweight component does it in `createPeer()`. The Canvas will spec the exact insertion point.

Asymmetric with AC10 by design; the Canvas will document the contract on both surfaces. |
| AC12 | Existing `eval` unchanged | Yes | New `evalAsync` is an additive method; the old `eval(String)` and its native call path are untouched. Regression risk is the addition of the resolver binding registration into `WebView.show()` and the engine `createPeer`/`addNotify` paths; these run only at peer-create time and do not affect later `eval` calls. |
| AC13 | Unserializable result → exceptional | Yes | `JSON.stringify(document.body)` typically returns a string for a DOM node (not throws — JSON.stringify of a plain DOM object is `{}` because DOM properties are non-enumerable, or it throws on circular ref). Either way, the wrapper's `.then` either resolves with the (possibly degenerate) string or the `.catch` fires for the circular-ref `TypeError`. **Refinement**: AC13's expectation that this *must* complete exceptionally is too strong — `JSON.stringify(document.body)` may simply yield `"{}"` (success), depending on engine. The Canvas should soften this AC: "completes exceptionally OR with a JSON string that does not faithfully represent the DOM node; the host's app does not crash and no native callback leaks." Worth raising with the user before the Canvas update. |
| NFR1 | No measurable eval-latency regression | Yes | Wrapper is small (single-digit lines); base64 encode of a tiny payload is microseconds; one `ConcurrentHashMap.put` per call. |
| NFR2 | In-flight map does not grow unbounded | Yes — with caveat | Every binding callback removes its id from the map. Cancelled or never-resolving futures (E2, E7) leak entries until dispose. Documented as known limitation. |
| NFR3 | Resolver binding registered once per peer lifetime | Yes | Eager registration at peer creation (R3). |
| NFR4 | Dispose drains pending futures | Yes | `disposeAllPending` called from `EmbeddedWebView.dispose` / `OffscreenWebView.dispose` / `WebView.show` post-loop (R8). |

**Open question to raise with the user before `/spdd-prompt-update`** (in order of priority):

1. **AC13 softening**: should the Canvas spec the unserializable-result case as "future completes exceptionally OR with a degenerate string, but never crashes"? AC13's "must complete exceptionally" is engine-dependent.
2. **`JavaScriptEvalException`**: introduce a dedicated exception type for JS-side failures (sync throw + Promise rejection + serialization failure), or use a generic `RuntimeException` with the message? Recommendation: dedicated type, but worth confirming.
3. **R1 binding rename**: confirm rename from `__swv_evalResult__` to `__webview_eval_result__` so the existing runtime collision guard protects the new binding too.
