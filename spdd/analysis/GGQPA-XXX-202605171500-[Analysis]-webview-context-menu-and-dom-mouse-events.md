# SPDD Analysis: WebView Right-Click Context Menu + DOM Mouse Event Channel

## Original Business Requirement

(Excerpt — full text preserved verbatim in `requirements/[User-story-2]webview-context-menu-and-dom-mouse-events.md`.)

A single story, `STORY-002-001 — Right-Click Context Menu and DOM Mouse Event Channel for WebViewComponent`. Scoped as two layers landing together:

- **Layer 1 (event channel).** A reserved JS binding (`__webview_dom_event`) carrying `contextmenu` payloads from page to Java. Java surface: `WebViewMouseEvent` (button, modifiers, viewport/page/screen coordinates, `toComponentPoint`, `toScreenPoint`), `DomTarget` (tag/id/classes/attributes/linkHref/imageSrc/mediaSrc/isContentEditable/selectionText/pageUrl/frameUrl), and `WebViewMouseListener.contextMenuRequested(WebViewMouseEvent)`. New methods on `WebViewComponent`: `addWebViewMouseListener`, `removeWebViewMouseListener`, `setDefaultContextMenuEnabled(boolean)`, `isDefaultContextMenuEnabled()`. Auto-suppress browser-default menu while at least one listener is registered; explicit setter overrides.
- **Layer 2 (JPopupMenu helper).** `ca.weblite.webview.swing.WebViewContextMenu(Function<WebViewMouseEvent, JPopupMenu>)` with `attachTo(WebViewComponent)` / `detach()`. On each `contextMenuRequested`, invokes the builder; if it returns non-null, calls `popup.show(event.source(), p.x, p.y)` with `p = event.toComponentPoint()`; `null` suppresses the popup.

Must work in both `HEAVYWEIGHT` and `LIGHTWEIGHT` modes. All callbacks fire on the Swing EDT. Listeners added before display are remembered. Shim is idempotent across navigations. Reserved binding name protected by the existing `RESERVED_BINDING_PREFIX` check. 22 ACs + 4 non-functional expectations.

Explicit non-goals: other DOM mouse event kinds (`mousedown`/`mouseup`/`dblclick`/`dragstart`), native OS context menus, drag-and-drop, custom tooltips, programmatic right-click dispatch, re-injecting an "Inspect Element" item when default is suppressed, and adding the API to the standalone in-process `WebView` class.

## Domain Concept Identification

### Existing Concepts (from codebase)

- **`WebViewComponent`** (`src/ca/weblite/webview/swing/WebViewComponent.java`) — abstract base. Currently exposes `setUrl/getUrl`, `setDebug`, `addOnBeforeLoad`, `eval`, `addJavascriptCallback`, `dispatch`, `dispose`, `openDevTools`, console-listener accessors, and the constant `RESERVED_BINDING_PREFIX = "__webview_"`. The new mouse-listener / default-menu API attaches here and is implemented uniformly across both subclasses. The new `__webview_dom_event` binding name is covered by the existing reserved-prefix check (`WebViewHeavyweightComponent.java:152-157`, `WebViewLightweightComponent.java:390-395`) — story AC22 is trivially satisfied by an existing invariant; no new guard needed.
- **`WebViewHeavyweightComponent`** (`src/ca/weblite/webview/swing/WebViewHeavyweightComponent.java`) — heavyweight peer hosted on a child `Canvas`; native peer created lazily on the canvas's first paint via `EmbeddedWebView.attach(canvas, debug)` (`createPeer`, line 394). Already exhibits the buffered-replay pattern (pending URL / init scripts / bindings flushed after `attach`) that the new mouse-listener and the default-menu-enabled flag must reuse. Already wires the console bridge in `createPeer()` (lines 410-417) by registering `ConsoleDispatcher.SHIM_JS` via `embedded.addOnBeforeLoad` and binding `__webview_console__` via `embedded.addJavascriptCallback` — exactly the symmetric pattern this story needs for `__webview_dom_event`.
- **`WebViewLightweightComponent`** (`src/ca/weblite/webview/swing/WebViewLightweightComponent.java`) — offscreen engine wrapping `OffscreenWebView`. Mouse events arrive through AWT and are forwarded into WebKit via `engine.mouseButton`/`mouseMotion`/`mouseScroll` (lines 109-156). **Important**: right-click (`MouseEvent.BUTTON3`) is already forwarded into WebKit through `GdkInput.translateButton`, so the offscreen engine synthesises a native `button-press-event` with the GDK secondary button, which WebKit turns into a DOM `contextmenu` event in the page. The JS shim picks it up there — no new native plumbing on the lightweight side. The component also already installs `ConsoleDispatcher.SHIM_JS` + the `__webview_console__` binding in `addNotify()` (lines 209-216); the new shim wires in exactly alongside it.
- **`EmbeddedWebView`** (`src/ca/weblite/webview/EmbeddedWebView.java`) — JNI wrapper around the heavyweight peer. Public methods `addOnBeforeLoad`, `addJavascriptCallback`, `eval`, `dispatch` are everything the story needs; no JNI additions on the heavyweight side. **Critical**: callbacks bound via `addJavascriptCallback` fire on whatever native UI thread the engine runs on (GTK main on Linux, AppKit main on macOS, the WebView2 worker on Windows) — **not** the EDT. The new `WebViewMouseDispatcher` (mirroring `ConsoleDispatcher`) must hop to the EDT before fanning out.
- **`OffscreenWebView`** (`src/ca/weblite/webview/OffscreenWebView.java`) — JNI wrapper around the offscreen peer. Already exposes `addOnBeforeLoad`, `addJavascriptCallback`, `eval`, `dispatch` (lines 126-183) — the analysis for STORY-001-002 surfaced these as "missing" but they have since been implemented (the story shipped, see commit history). No JNI additions on the lightweight side either.
- **`ConsoleDispatcher`** (`src/ca/weblite/webview/ConsoleDispatcher.java`) — **the canonical template** for the new dispatcher. Established patterns to copy 1-for-1:
  - JS shim as a `public static final String SHIM_JS` constant on the dispatcher class, injected via `addOnBeforeLoad` at peer-create time.
  - Reserved binding name as a `public static final String CHANNEL_NAME` constant.
  - Re-entry guard at top of shim (`window.__webview_console_installed__`) so navigation re-runs are idempotent (matches AC21).
  - Wire format: base64-encoded UTF-8 string of a pipe-separated payload, posted as the binding's single string argument. Java side decodes via `Base64.getDecoder()` and `new String(bytes, "UTF-8")`. This deliberately sidesteps JSON-escape complexity — the new mouse payload will follow the same convention.
  - `CopyOnWriteArrayList<Listener>` registry with `addListener` / `removeListener` taking the existing semantics: a snapshot is captured at dispatch time, so add/remove during fan-out only takes effect for the next event.
  - EDT marshaling: `SwingUtilities.isEventDispatchThread()` short-circuit + `invokeLater`.
  - Listener exception isolation: each `listener.onMessage(msg)` is wrapped in `try { ... } catch (Throwable t) { uncaughtExceptionHandler.uncaughtException(...) }`. Story AC20 maps directly to this pattern.
- **`ConsoleListener`** (`src/ca/weblite/webview/ConsoleListener.java`) — single-method functional interface. The new `WebViewMouseListener` matches this shape, but has an extension axis: future event kinds (`mousePressed`, `dblclick`, `dragstart`, etc.) must be addable without breaking binary compatibility. Java 8 default methods on the interface are the chosen mechanism — `contextMenuRequested` is the only required method now; future kinds will be `default` no-ops.
- **`WebViewClickCallback`** (`src/ca/weblite/webview/WebViewClickCallback.java`) — existing coarse mouse-press hook on the heavyweight peer. Only fires that *some* button was pressed inside the WebView and is used internally by `WebViewHeavyweightComponent` (lines 449-460) to drive `MenuSelectionManager.defaultManager().clearSelectedPath()` — the workaround for AWT's `MouseGrabber` not seeing presses that land on heavyweight native peers. The story's new mouse listener does **not** subsume this: `WebViewClickCallback` is package-private to the embedded path, exists for popup-dismissal only, and runs whether or not any Java mouse listener is registered. Both coexist; no changes to the click callback in this story.
- **JS-binding bridge** — every engine already routes binding calls through a `{name, seq, args}` JSON wrapper, with the first arg being whatever the page passed. The `ConsoleDispatcher.extractFirstArg(json)` helper (lines 255-263) parses this wrapper without a full JSON parser (relying on the base64 encoding to keep the inner string JSON-special-character-free). The same wrapper format is used by the new `__webview_dom_event` channel.
- **`addOnBeforeLoad`** lifecycle — accumulates in a `pendingInit: List<String>` until peer-create; on `createPeer()` (heavyweight) / `addNotify` (lightweight), each entry is replayed via `embedded.addOnBeforeLoad` / `engine.addOnBeforeLoad`. **Notably**: `addOnBeforeLoad` is append-only — there is no "remove this init script" or "replace this init script" operation on either engine. The story's runtime suppression-flag toggling must work around this (see Risk & Gap Analysis).
- **`eval`** lifecycle — fires JS against the *current* document. Used here to flip the suppress flag on the live page when a listener is added/removed mid-session. No-op until the peer is displayable.
- **`RESERVED_BINDING_PREFIX`** (`WebViewComponent.java:51`) — the string `"__webview_"`. Both subclasses' `addJavascriptCallback` reject names starting with this prefix (`IllegalArgumentException`). The new `__webview_dom_event` channel name is automatically protected; story AC22 needs no new code.

### New Concepts Required

- **`ca.weblite.webview.WebViewMouseEvent`** — immutable value object carrying one DOM mouse event's worth of data: event type (string, fixed to `"contextmenu"` in this story), button (1/2/3), modifier-key state (shift/ctrl/alt/meta), three coordinate frames (viewport `clientX/Y`, document `pageX/Y`, screen `screenX/Y`), `timeStamp` (ms since navigation start), and references to `target: DomTarget` and `source: WebViewComponent`. Convenience helpers `toComponentPoint()` and `toScreenPoint()` returning `java.awt.Point` for `JPopupMenu.show`.
- **`ca.weblite.webview.DomTarget`** — immutable descriptor of the element the user right-clicked. Tag (uppercase), id, class list, curated attribute map (subset of `href / src / alt / title / name / type / value / role / data-*`), nearest-ancestor link `href`, image `src` / media `src` when relevant, `isContentEditable`, current selection text, page and frame URLs.
- **`ca.weblite.webview.WebViewMouseListener`** — single-required-method interface in this story (`contextMenuRequested`). Designed for forward extension via `default` methods.
- **`ca.weblite.webview.WebViewMouseDispatcher`** — per-component fan-out hub mirroring `ConsoleDispatcher`. Owns the `CopyOnWriteArrayList<WebViewMouseListener>`, owns the `boolean defaultContextMenuEnabled` flag, owns the `WebViewComponent source` reference needed to construct events, exposes the canonical `SHIM_JS` string and the `CHANNEL_NAME` constant, and implements `dispatch(String rawJson)`. Like `ConsoleDispatcher`, internal-only — not part of the public API surface — but `public` because consuming Swing subclasses live in a different package (same constraint the existing `ConsoleDispatcher` documents at the top of its Javadoc).
- **`ca.weblite.webview.swing.WebViewContextMenu`** — Swing-package convenience helper. Owns a `Function<WebViewMouseEvent, JPopupMenu>` builder, registers an internal `WebViewMouseListener` on `attachTo`, removes it on `detach`. On each event: invoke builder; if non-null, `popup.show(event.source(), point.x, point.y)`.
- **Reserved binding name `__webview_dom_event`** — the channel through which JS posts mouse-event payloads. Automatically protected by `RESERVED_BINDING_PREFIX`.
- **JS shim for `contextmenu`** — a new `public static final String SHIM_JS` constant on `WebViewMouseDispatcher`. Installs once per page via a `window.__webview_dom_event_installed__` guard. Adds a `document.addEventListener('contextmenu', ...)` capture-phase handler that:
  1. Reads `window.__webview_dom_event_suppress` and calls `event.preventDefault()` when it's truthy.
  2. Builds the canonical pipe-separated payload (event fields + descriptor fields, with classes / attributes / selection text base64-encoded individually to preserve characters that would otherwise collide with the field separator).
  3. Base64-encodes the whole payload as a single UTF-8 string and posts via `window.__webview_dom_event(b64)`.
- **Default-menu suppression flag transport** — a JS-side `window.__webview_dom_event_suppress` variable. Java keeps its own `boolean suppress` and mirrors it via `eval()` whenever the state changes (listener added/removed/`setDefaultContextMenuEnabled` called). See Risk & Gap Analysis for the cross-navigation persistence question.

### Conceptual Relationships

- `WebViewComponent` *owns* a `WebViewMouseDispatcher` for its entire lifetime, in exactly the way it owns a `ConsoleDispatcher` today (the dispatcher field is `protected final` on the base class and initialized at construction).
- The dispatcher *owns* the listener registry and the suppression flag. Adding the first listener implicitly suppresses the default menu; removing the last listener implicitly restores it. `setDefaultContextMenuEnabled(boolean)` is an explicit override that overrides the implicit value for as long as it stays in effect.
- Pre-display listener registration is buffered the same way URL / init scripts / bindings / console listeners are buffered — the dispatcher's listener list is populated from creation onward, but the JS shim and binding only get installed when the native peer attaches. Events that fire before any listener is registered are dropped (the JS shim isn't there to catch them; the page just shows the platform default menu).
- The shim binding `__webview_dom_event` rides the same per-engine native message bridge used by `__webview_console__` and any user-defined binding — there's no resource contention because each binding name is a distinct script-message-handler on each platform.
- `WebViewContextMenu` *uses* the dispatcher transparently (through the public `addWebViewMouseListener` API) — it has no privileged access. This means callers can mix `WebViewContextMenu` with their own direct listeners and they all fire in registration order (story AC19's contract holds across helper-internal and caller-direct listeners alike).
- The new dispatcher is independent of `ConsoleDispatcher` — they share patterns but not state. Both can be installed simultaneously without interference; each registers its own `addOnBeforeLoad` and its own binding.
- `openDevTools` (from STORY-001-001) is orthogonal — the story Scope-Out's "Inspect Element re-injection" risk only matters because suppressing the platform menu removes the right-click → Inspect Element entry point on macOS. The Java-side `openDevTools()` call is unaffected; callers can still add an "Open DevTools" item to their own `JPopupMenu` as a workaround.

### Key Business Rules

- **Reserved-name discipline.** Callers cannot bind a JS callback under the name `__webview_dom_event` (or any name starting with `__webview_`). The existing `addJavascriptCallback` check enforces this — story AC22 needs zero new code.
- **EDT-only callbacks.** Every `contextMenuRequested` invocation runs on the Swing EDT, so listeners may touch Swing state directly. The dispatcher uses `SwingUtilities.invokeLater` (matching `ConsoleDispatcher`'s contract). Story AC17 maps directly.
- **One JS shim per peer-attach.** The shim is added via `addOnBeforeLoad`, which guarantees it runs at every new document load — no Java-side re-installation across navigation. The shim's `window.__webview_dom_event_installed__` guard makes repeat invocations during a single document idempotent. Story AC21 maps directly.
- **Auto-suppress on first listener; auto-restore on last removal.** This is a stateful policy: the dispatcher tracks listener count and flips the JS-side suppress flag on the 0↔1 transitions (and on every `setDefaultContextMenuEnabled` call). Story ACs 10, 11, 12, 13, 16 all derive from this rule.
- **Pre-existing reserved-binding invariant.** This story does not introduce a new reserved-prefix policy — it relies on the policy already declared by STORY-001-002 (`RESERVED_BINDING_PREFIX` on `WebViewComponent`).
- **No `setDebug` coupling.** The default-menu suppression works in release builds (debug=false). The shim is content-script-style and does not require developer extras to be enabled. (Contrast with `openDevTools`, which *does* require debug.)
- **Password input scrubbing.** When the right-clicked element is an `<input type="password">`, its `value` attribute is reported as the empty string regardless of actual content. Story NF expectation maps directly; enforced in the JS shim's descriptor builder.
- **Sensitive selection text.** The story does NOT mandate scrubbing selection text from password fields (the page's own `window.getSelection().toString()` would return empty anyway because password-field selection is opaque to the DOM). The risk surface is therefore implicit; no extra code needed.

## Strategic Approach

### Solution Direction

**Replicate the `ConsoleDispatcher` pattern for a second, parallel channel — `WebViewMouseDispatcher` + reserved `__webview_dom_event` binding — and add a tiny Swing-side helper on top.**

For Layer 1 (event channel):
- Add `WebViewMouseEvent`, `DomTarget`, `WebViewMouseListener` to `ca.weblite.webview` as public types. All immutable value objects / functional interfaces — no JNI, no native, no Swing dependencies.
- Add `WebViewMouseDispatcher` to `ca.weblite.webview` following `ConsoleDispatcher` line-for-line: `public static final String SHIM_JS`, `public static final String CHANNEL_NAME = "__webview_dom_event"`, `CopyOnWriteArrayList<WebViewMouseListener>`, `dispatch(String rawJson)` that decodes the payload and fans out on the EDT with try/catch isolation per listener.
- Add four methods to `WebViewComponent` (abstract on the base, identically delegating implementations on the two subclasses): `addWebViewMouseListener`, `removeWebViewMouseListener`, `setDefaultContextMenuEnabled`, `isDefaultContextMenuEnabled`. Each delegates to the dispatcher's per-component instance. Listener-count transitions (0→1 and 1→0) trigger a re-evaluation of the JS-side suppress flag.
- Both subclasses install the shim + binding in their existing peer-bring-up paths — `createPeer()` in heavyweight (right after the console bridge install, lines 410-417), `addNotify()` in lightweight (right after the analogous lines 209-216). Mirrors the console-capture sibling exactly.
- The dispatcher's JS shim is added via the peer's `addOnBeforeLoad` (bypassing the public `addJavascriptCallback` so the reserved-prefix check doesn't reject our own internal binding — same trick used for `__webview_console__`).

For Layer 2 (JPopupMenu helper):
- Add `WebViewContextMenu` to `ca.weblite.webview.swing` as a small standalone class. Constructor takes the menu builder `Function<WebViewMouseEvent, JPopupMenu>`. `attachTo(WebViewComponent)` calls `addWebViewMouseListener` with an internal listener that invokes the builder and shows the result; `detach()` removes it. Roughly 40-60 lines.
- No new state on `WebViewComponent` for this layer — the helper is pure composition over the Layer 1 API.

JS shim implementation strategy:
- Capture-phase `contextmenu` handler at `document` level. Reads `window.__webview_dom_event_suppress`; calls `preventDefault()` when truthy.
- Builds descriptor by walking up from `event.target`: extract tag / id / classes / curated attribute set, find nearest `<a>` for `linkHref`, find `<img>`/`<video>`/`<audio>` for `imageSrc`/`mediaSrc`, read `target.isContentEditable` (and check for `INPUT`/`TEXTAREA`), read `window.getSelection().toString()` for `selectionText`, read `document.location.href` for `frameUrl` and find the top-level document for `pageUrl`. For each curated attribute, special-case `<input type="password">` to scrub `value`.
- Wire format: a single base64-encoded UTF-8 string containing a versioned, pipe-separated record. Inner fields that may contain pipes or newlines (selection text, attribute values, class names) are themselves base64-encoded into individual sub-fields before the outer base64 wraps the whole thing. Java side decodes the outer base64, splits on pipe, and decodes each base64 sub-field. **This avoids embedding a full JSON parser into Java for what is at most a few-hundred-byte payload.** Same constraint that drove `ConsoleDispatcher`'s wire format.
- Re-entry / install guards: `if (window.__webview_dom_event_installed__) return;` then `window.__webview_dom_event_installed__ = true;`. Matches the console shim's idempotency pattern.

Default-menu suppression-flag transport:
- Java keeps `boolean defaultEnabled` per dispatcher (default `true`, since with zero listeners the platform default wins).
- The *effective* suppression state at any moment is `!defaultEnabled && hasListeners()`.
- Whenever the effective state changes — listener added/removed crosses the 0↔1 boundary, or `setDefaultContextMenuEnabled` is called — the dispatcher calls `webview.eval("window.__webview_dom_event_suppress=" + effective + ";")` on the live page, which takes effect for the next contextmenu event in the current document.
- For cross-navigation persistence: see Risk & Gap Analysis below — the proposed resolution is to additionally register a tiny `addOnBeforeLoad` snippet `"window.__webview_dom_event_suppress=<currentValue>;"` whenever the effective state changes, accepting that the engines' append-only `addOnBeforeLoad` means later overrides supersede earlier ones at document-start (last-assignment-wins in the JS execution order).

Coordinate translation strategy:
- The JS event reports CSS pixels. AWT reports Java component coordinates in *logical* pixels (which, on modern Swing with HiDPI, match CSS logical pixels 1:1 because both abstract over the device pixel ratio).
- `toComponentPoint()` returns `new Point(clientX, clientY)` directly — the WebView component's content area corresponds 1:1 with the page's viewport, so viewport CSS pixels = component logical pixels. No DPI math required, and the story's "no new HiDPI handling" Scope-Out is naturally satisfied.
- `toScreenPoint()` returns `new Point(screenX, screenY)` directly — `event.screenX/Y` is the standard DOM screen coordinate (CSS pixels relative to the screen, accounting for window position).

### Key Design Decisions

- **Mirror `ConsoleDispatcher` vs. invent a new dispatcher pattern.** Trade-off: copying produces a second class with ~70% the structure of `ConsoleDispatcher` (CopyOnWriteArrayList + EDT marshaling + try/catch + base64 wire format) but tailored for a different payload. Extracting a shared base class would eliminate ~50 lines of duplication. **Recommendation: copy, don't generalize.** The two payloads differ enough (single line vs. multi-field record) that a shared base would need generics or a `Function<String, Event>` parser injection — more complexity than the duplication costs. The codebase already has two precedents (`EmbeddedWebView` and `OffscreenWebView` are deliberately parallel rather than sharing a base) so the pattern is house-style.

- **Single binding `__webview_dom_event` vs. per-event-kind bindings.** Trade-off: a single channel transports payloads tagged with an event-type field, even though only `"contextmenu"` ships now. Per-kind bindings (`__webview_contextmenu`, future `__webview_mousedown`, etc.) would be slightly easier to reason about. **Recommendation: single channel.** Forward-compatible with adding more event kinds without new bindings or new RESERVED_BINDING_PREFIX entries; matches the design choice already made for `__webview_console__` (single channel for five console levels).

- **Java 8 `default` methods on `WebViewMouseListener` for future event kinds vs. multiple listener interfaces.** Trade-off: `default` methods preserve binary compatibility — callers writing today's listener with only `contextMenuRequested` won't break when `mousePressed` is added. Multiple interfaces (e.g. `ContextMenuListener` extends `WebViewMouseListener`) would let callers opt into specific kinds, but the API surface grows linearly with each new kind. **Recommendation: one interface with `default` methods.** Library targets Java 8 (`pom.xml:39-41`); the language feature is fully available.

- **Wire format: base64-encoded pipe-separated record vs. JSON.** Trade-off: pipe format requires writing a parser on the Java side (or careful escaping); JSON would let us use any Java-side JSON library. **Recommendation: pipe-separated, base64-protected.** The library has no JSON dependency today and the analysis for STORY-001-002 explicitly chose this format to avoid adding one. The new dispatcher reuses `ConsoleDispatcher.extractFirstArg` as a model and stays dependency-free.

- **Suppression-flag transport: eval-only vs. eval + addOnBeforeLoad.** Trade-off: pure `eval` is simple but the flag value resets on every navigation (the shim's hardcoded default takes over). `eval` + a per-state-change `addOnBeforeLoad` makes the flag survive navigation by re-asserting at document-start, at the cost of an accumulating list of `addOnBeforeLoad` entries (each engine has no removal API). **Recommendation: eval + addOnBeforeLoad.** Each state change appends one short statement (`"window.__webview_dom_event_suppress=true;"`); in steady state the page accumulates a small constant number of these (typical caller flips the state maybe two or three times in a session). Memory and parse-time cost is negligible.

- **JPopupMenu show coordinate frame: `toComponentPoint` vs. `toScreenPoint`.** Trade-off: `JPopupMenu.show(Component, int, int)` takes component-relative coordinates; `JPopupMenu.show(Component, int, int)` with a screen-relative interpretation requires `null` for the component or extra Swing gymnastics. **Recommendation: `toComponentPoint`.** Matches Swing's idiomatic call shape, makes story AC14 ("within one pixel of (120, 80) in the component's coordinate space") directly satisfiable, and avoids any cross-window-frame surprises.

- **Auto-suppression policy lock-in.** The story's "auto-suppress when listener registered" decision is final per the requirements. **Recommendation: document this clearly in the canvas Safeguards as a deliberate API choice**, so a future maintainer doesn't "fix" it by making suppression opt-in (which would silently introduce double-menu bugs in every caller's first iteration).

- **Heavyweight JPopupMenu painting over native WebView.** This isn't a new decision but an *inherited constraint*: heavyweight native peers paint above lightweight Swing popups. Both the heavyweight demo (`demos/WebViewHeavyweightDemo.java:83-84`, `159-161`) and the console demo (`demos/WebViewConsoleDemo.java:41-42`) globally call `JPopupMenu.setDefaultLightWeightPopupEnabled(false)` and `ToolTipManager.sharedInstance().setLightWeightPopupEnabled(false)` to force heavyweight popups so they paint above the native WebView. **Recommendation: document this prerequisite in the canvas Norms** and either (a) have `WebViewContextMenu.attachTo` defensively set the JVM-global flags itself (with a property override to opt out), or (b) just warn callers in Javadoc. Decision deferred to canvas — leaning (b) because (a) mutates global Swing state from a constructor-level call.

### Alternatives Considered

- **Platform-native context menus (NSMenu / GtkMenu / HMENU).** Rejected by the user up-front: explicitly a non-goal for this story. The native path would require three per-platform implementations, three sets of menu-builder APIs, and a coordinate / IME / theming story that doesn't compose with Swing. Swing JPopupMenu is consistent across all three platforms and integrates with Swing actions the host app already has.

- **Use each platform's native context-menu interception API instead of JS preventDefault.** WebView2 exposes `ICoreWebView2_4::add_ContextMenuRequested` (Windows) and `WKUIDelegate::webView:contextMenuConfigurationForElement:completionHandler:` (macOS 13.3+) for native interception with element metadata. Rejected because (a) WebKitGTK has no equivalent native API (we'd need JS preventDefault anyway, so we'd be maintaining two paths), (b) JS preventDefault works uniformly across all three platforms today, (c) the DOM target descriptor is richer than what the native APIs expose. Noted as a future optimisation for Windows/macOS if richer integration (e.g. native sub-menus mixed with Java menus) is ever needed.

- **`MouseListener` (AWT) extension instead of new interface.** Considered using AWT's existing `MouseListener` / `MouseEvent` for the contract. Rejected because (a) AWT `MouseEvent` doesn't carry DOM target info, so we'd be subclassing it which is brittle, (b) `MouseListener.mouseClicked` semantics differ from web `contextmenu` semantics, (c) the descriptor object (`DomTarget`) is the actual value-add and bolting it onto AWT classes would be awkward. The new interface costs nothing and reads more clearly.

- **Surface the JS event payload directly to callers (no `WebViewMouseEvent` POJO).** Rejected: callers should not have to parse the wire format. The POJO is the public API contract; the wire format is implementation detail.

- **Synchronous JS round-trip to ask Java whether to suppress.** I.e. let Java decide per-event whether to call preventDefault. Rejected: synchronous JS↔Java calls are deeply platform-specific (and outright impossible without blocking the engine thread on WebView2). The current "Java sets the flag, JS reads it" model is async-friendly and has no measurable latency cost.

- **Combine `addWebViewMouseListener` and `WebViewContextMenu` into a single helper API.** Rejected: keeping them separate respects layering. Callers who want raw events get them; callers who want a JPopupMenu helper get that; future callers wanting (say) drag-and-drop or tooltips will still need the raw listener API even after the helper is shipped.

## Risk & Gap Analysis

### Requirement Ambiguities

- **Cross-navigation persistence of the `setDefaultContextMenuEnabled(true)` override.** The story's NF expectation says "Toggling `setDefaultContextMenuEnabled(boolean)` must take effect on the very next right-click (no stale-flag invocations after the setter returns)." This unambiguously covers the same-document case. It does not explicitly say the override survives navigation. **Resolution direction**: the canvas should commit to "the override survives navigation" because (a) it matches caller intent (a setting on the component shouldn't reset under their feet), (b) it's implementable via the eval + addOnBeforeLoad approach above without much extra work, (c) the alternative (silently reverting on every navigation) would be surprising. Lock in the canvas Safeguards.
- **`event.timeStamp` origin.** Story says "milliseconds since navigation start". The DOM `Event.timeStamp` on modern engines is a `DOMHighResTimeStamp` (a `double`) whose origin is the time origin (typically navigation start of the top-level document) — fractional milliseconds, monotonic, not wall-clock. **Resolution direction**: surface the `long` cast to Java by truncating the fractional part. Document "monotonic, milliseconds since the top-level document's time origin"; do not promise wall-clock semantics.
- **Multiple right-clicks in a single user gesture (e.g. mac trackpad two-finger tap that resolves to BUTTON3 + immediate BUTTON1 release).** Story is silent on whether spurious extra events should fire. **Resolution direction**: trust the page-side `contextmenu` event semantics — one DOM `contextmenu` event = one Java callback. The browser engines already de-duplicate platform input gestures down to one `contextmenu` event per logical right-click. No Java-side de-duplication needed.
- **Very large selections.** `window.getSelection().toString()` could return megabytes of text. **Resolution direction**: cap selection text at a sane limit (proposed: 64 KiB UTF-8) in the JS shim, append `...` if truncated. Lock the limit in the canvas; mention in Javadoc.
- **`data-*` attribute payload size.** Some pages decorate elements with many `data-*` attributes (e.g. analytics frameworks). The story says `data-*` keys "present on the element" are surfaced — but a single element could have dozens of `data-*` attributes summing to many KB. **Resolution direction**: cap total `data-*` payload at a sane limit (proposed: 8 KiB total across all `data-*` keys), drop overflow with a documented overflow marker. Lock in the canvas.
- **Drop-listeners semantics (events that fire before any listener exists).** Story says "events that fire before any Java listener is registered are dropped." Confirmed: the dispatcher simply doesn't install the JS shim until the first listener is added, so the page emits its native default-menu and no callback fires. No ambiguity.

### Edge Cases

- **JPopupMenu painting under heavyweight native WebView.** As noted in the Approach section, the heavyweight peer paints above lightweight Swing popups. **Caller must** set `JPopupMenu.setDefaultLightWeightPopupEnabled(false)` (or use `setLightWeightPopupEnabled(false)` on individual popups) so JPopupMenu uses a heavyweight top-level. The story doesn't address this. **Resolution direction**: canvas Norms should document this prerequisite prominently. AC14 will fail under the default lightweight-popup setting on heavyweight components; this is a precondition, not a defect in the implementation.
- **Right-click on `<select>` dropdown or any element that has its own native UI.** Browsers may suppress the `contextmenu` event entirely for certain native controls (e.g. video element controls). **Resolution direction**: this is intrinsic to the platform — when no `contextmenu` event fires in JS, the Java side gets nothing. Document as an expected non-occurrence.
- **Right-click during JS-driven page reload / between unload and load.** The shim isn't available during the load gap; the page's default menu may fire. **Resolution direction**: ignorable in practice — this is a millisecond-scale window the user can't reliably hit. Don't add code to handle it.
- **Shim runs in a Trusted Types-enforcing CSP context.** CSP can block inline scripts via `script-src 'self'`. The library injects via the engine's "execute script on document creation" API (WKUserScript / WebKitUserScript / WebView2 AddScriptToExecuteOnDocumentCreated) which is exempt from CSP. **Resolution direction**: same as the console shim, which has the same protection. No new risk.
- **`document.designMode = 'on'` or full-page contenteditable.** All elements are then content-editable. `DomTarget.isContentEditable` will be `true` for any element. **Resolution direction**: this is the correct semantics. Document in DomTarget Javadoc.
- **Same-origin iframe.** Story scope says same-origin iframes are supported with `frameUrl != pageUrl`. To make this work, the shim must be installed in every same-origin iframe (or use a `document` listener at the top window that catches bubbled events from same-origin children). Modern engines' `addScriptToExecuteOnDocumentCreated`-equivalent injects into every same-origin document by default. **Resolution direction**: rely on the engine's default per-document injection, plus a top-level capture-phase `contextmenu` listener catches all bubbled events. Verify in canvas that the install API does per-document injection on all three platforms.
- **Cross-origin iframe.** Story Scope-Out says only the iframe element itself is reported. **Resolution direction**: the top-level `contextmenu` listener won't fire for events that originate inside the cross-origin iframe (the event doesn't cross the origin boundary). The Java side sees no event at all. Document as the expected behaviour.
- **`<input type="password">` `value` scrubbing.** The shim must replace the password input's `value` attribute with `""` before serializing. **Resolution direction**: explicit check on `target.tagName === 'INPUT' && target.type === 'password'` — drop `value` from the curated attribute set.
- **Right-click on the page when the page itself has `addEventListener('contextmenu', e => e.preventDefault())`.** The page's listener may run before the library's (depending on capture vs. bubble). **Resolution direction**: install the library shim at the capture phase on `document` so it runs *before* any bubble-phase page listener. The library always sees the event. Story AC10 still works because the *platform default* menu doesn't appear (regardless of which side called preventDefault).
- **Concurrent `setDefaultContextMenuEnabled` from non-EDT threads.** The dispatcher's flag is a single `volatile boolean` and the eval call goes through the engine's thread-safe `eval` (no special handling required). **Resolution direction**: declare the field `volatile` and don't synchronize beyond that.

### Technical Risks

- **`addOnBeforeLoad` accumulation for the suppress-flag override.** Each `setDefaultContextMenuEnabled` call appends one short statement. In a 24-hour kiosk session that flips state every minute, that's ~1500 statements appended. Each is ~50 bytes; total payload ~75 KB which is fine but not free. **Mitigation direction**: in practice callers flip 2-3 times per session, so this is a non-issue. If it ever becomes one, the canvas can introduce a "consolidate" path that compacts the override at significant transitions.
- **Race between `eval` of suppress flag and a fast right-click that arrives before eval completes.** `eval` is async on all three engines. A right-click that arrives in the millisecond between Java calling `setDefaultContextMenuEnabled(true)` and the engine actually applying the flag would use the old value. **Mitigation direction**: accept this as a documented edge case ("very-next right-click" in the NF expectation could fail under a sub-ms race). The user's gesture latency is already two-three orders of magnitude slower than the eval latency, so this is theoretical.
- **`WebViewLightweightComponent` on non-Linux platforms.** The offscreen engine returns `null` from `create()` on macOS/Windows, so `addNotify` short-circuits and never installs the shim. ACs that target the lightweight component will not be testable on those platforms. **Mitigation direction**: this is an existing limitation documented in `WebViewLightweightComponent.java:62-68` and the canvas-7 norms. Story AC1 is Linux-specific anyway; story AC14's "WebViewComponent" can be tested with either subclass on platforms where they work. Document in canvas.
- **macOS WKWebView 12.x and earlier `event.screenX/Y` reliability.** Some older WKWebView versions report `screenX/Y` as `clientX/Y` (a long-standing WebKit bug). **Mitigation direction**: surface in canvas Safeguards. If `toScreenPoint()` is unreliable on those platforms, callers can fall back to converting via `SwingUtilities.convertPointToScreen(toComponentPoint(), source())`.
- **Listener exception swallowing semantics.** The existing `ConsoleDispatcher` catches `Throwable` and forwards to `Thread.getDefaultUncaughtExceptionHandler()`. Story AC20 says "surfaced via standard EDT uncaught-exception handling". **Mitigation direction**: match the existing pattern verbatim — same try/catch, same default-handler dispatch.
- **Reference-counting the JS shim across mixed feature usage.** Future stories may add more shim-using features. Each will have its own `addOnBeforeLoad` install — there is no shared registry. **Mitigation direction**: not a concern for this story (only console + DOM event shims exist), but the canvas should mention that the install is one-shot per feature and there is no cross-feature coordination.
